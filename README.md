# VC Deep Research

这个 skill 用来做单家公司深度研究。它适合在已经通过初筛找到目标公司后，进一步判断这家公司是否值得投入更多时间、资源和关系。

它回答一个核心问题：这家公司是否值得继续看，为什么。

这个 skill 的默认读者是买方投资团队。写作目标不是帮公司讲故事，而是测试公司的产品、需求、付款逻辑、监管准备度、中国落地路径和 VC 相关性是否经得住现实检查。

## 适用场景

当用户已经给出一家公司，并希望做投资判断或深度尽调时使用。

典型请求包括：

- 深挖这家公司
- 帮我做一篇公司深度研究
- 这家公司值不值得看
- 帮我判断它有没有投资价值
- 看看它在中国有没有落地机会
- 做一份单公司投资 memo

这个 skill 常接在 `Ophthalmic Company Search` 之后使用。前者负责发现公司，后者负责判断公司。

## 不适用场景

不要用这个 skill 处理以下任务：

- 广泛找公司
- 行业入门教育
- 多家公司竞品全景图
- 创始人宣传稿
- 泛泛的市场规模总结
- 没有指定公司的研究请求

如果用户只是想找一批欧洲或北欧眼科公司，应改用 `Ophthalmic Company Search`。

## 核心判断标准

Exactitude is the standard.

所有关键判断都必须区分三类信息：

- Confirmed：来源能直接支持的事实
- Inference：基于事实作出的推断
- Not verified：没有找到可靠证据的信息

不要把公司说法当成研究结论。不要把试点、合作、医院标志、会议露出自动当成商业化证据。不要用市场规模掩盖买方逻辑薄弱的问题。

研究时必须分清：

- 有趣的技术
- 临床上有用的产品
- 客户真的能买到的产品
- 能规模化销售的业务

一家公司可以技术很漂亮，但商业路径很薄。这个 skill 要把这种情况讲清楚。

## 研究优先级

按以下顺序判断，不要一上来写市场规模：

1. 问题是否真实
2. 买方是否真实
3. 付款逻辑是否真实
4. 产品是否足够成熟
5. 差异化是否有意义
6. 市场是否足够大且可触达
7. 中国落地是否可信
8. VC 是否真的有角色

如果买方和付款逻辑不成立，市场再大也不能支撑正向结论。

## 证据来源

优先找能直接验证公司状态的材料：

- 公司官网、产品页、白皮书、用户案例
- 产品说明书、监管注册、认证文件
- 临床试验、论文、会议摘要
- 年报、公告、投资者关系材料
- 融资新闻、投资机构公告、数据库记录
- 医院、大学、政府项目、采购和合作页面
- 专利、监管、报销、渠道和商业合作信息

对每个重要事实都要说明证据状态。信息缺失时直接写“无明确信息”或“未找到可靠证据”。

## 最终报告结构

所有正式报告必须用中文输出，并按固定顺序写。

开头先给一段判断，回答：

- 是否值得继续看
- 真正原因
- 最大风险
- 最可能改变判断的缺失证据

然后按以下章节展开：

0. 产品与市场入门指南
1. 公司定义
2. 应用场景
3. 客户群体
4. 商业模式
5. 技术、临床和监管准备情况
6. 差异化与护城河
7. 相关市场规模
8. 中国落地路径
9. 最终建议

最终建议必须使用以下结构：

```text
Recommendation: Proceed / Watch / Pass

Core reason:

Biggest open question:

Next thing that must be verified:
```

## 各章节要点

### 0. 产品与市场入门指南

用外行能懂的话讲清楚产品是什么、解决什么疾病或流程问题、为什么以前难解决、核心技术如何产生价值。必须解释核心术语，眼科公司必须解释相关眼部结构和疾病机制。

### 1. 公司定义

用最短且准确的方式定义公司。写清楚它实际卖什么、属于什么真实类别、用于什么场景、谁是最具体的使用者。融资信息要拆成 Confirmed、Partially reported、Unclear / unavailable。

### 2. 应用场景

判断真实 workflow 问题。说明现在谁痛、现有方案是什么、现有方案哪里不够好、痛点是成本、速度、准确性、人力、准入、合规、转诊效率还是收入转化。

痛点强度分为：

- nice-to-have improvement
- meaningful workflow improvement
- must-solve pain point

### 3. 客户群体

必须拆分角色：

| 角色 | 具体对象 | 证据状态 | 说明 |
|---|---|---|---|
| End user |  | Confirmed / Inference / Not verified |  |
| Economic buyer |  | Confirmed / Inference / Not verified |  |
| Procurement approver |  | Confirmed / Inference / Not verified |  |
| Beneficiary |  | Confirmed / Inference / Not verified |  |
| Channel partner |  | Confirmed / Inference / Not verified |  |

重点不是“谁可能喜欢”，而是谁使用、谁付钱、谁审批、谁受益，以及这个链条能不能重复销售。

### 4. 商业模式

判断公司卖的是设备、软件、服务、工作流、结果，还是组合。写清楚收入模式、价格证据、销售周期、交付复杂度、服务负担、是否依赖试点或一次性项目。

合作案例要区分商业部署、试点、临床研究、科研合作、分销协议、共同开发、资助项目和不明确关系。

### 5. 技术、临床和监管准备情况

判断技术是否已经产品化，是否可部署，是否有临床或验证证据，是否有监管批准。医疗和眼科公司必须说明技术、临床、监管三个层面的成熟度，不要把其中一个层面的强证据替代另一个层面。

### 6. 差异化与护城河

不要强行写护城河。先判断真正不同的地方是什么，再判断买方是否在意，最后判断严肃竞争者能否复制。

差异可以来自技术、临床、监管、工作流、渠道、成本、品牌、数据、服务或地理位置。只有能影响购买决策的差异才有商业意义。

### 7. 相关市场规模

选离公司业务最近的指标，不要机械引用大行业规模。医疗、眼科、筛查和诊断项目优先使用疾病人数、确诊人数、年度筛查量、相关手术量、相关机构数量或具体设备市场规模。

每个数字都要写清楚地理范围、年份、来源、单位、数字含义，以及它是直接相关还是仅作背景。

### 8. 中国落地路径

这一节不是最终投资建议，而是测试最小可信中国切入点和 VC 相关性。

必须写清楚：

- 中国现状和替代方案
- 最具体的中国用户、买家、审批者和使用场景
- 最小可售单位
- 已观察到的可售性证明
- 可能的中国落地路径
- VC 能不能减少搜索成本、信任成本或执行风险
- 主要未知数

不要只写“通过合作伙伴进入中国”。必须说明什么类型的合作伙伴，卖给谁，用于什么场景。

## 推荐分级

最终 Recommendation 只能选三种：

| Recommendation | 使用条件 |
|---|---|
| Proceed | 问题真实，买方清楚，付款逻辑基本成立，产品成熟度足够，中国切入点或 VC 角色有实际抓手 |
| Watch | 产品或临床逻辑有意思，但商业化、付款、中国路径或证据仍不足 |
| Pass | 痛点弱、买方不清楚、付款逻辑弱、差异化弱，或中国和 VC 相关性不成立 |

推荐必须从前文分析自然推出。不能先定结论，再找材料填空。

## 写作标准

写作要像会影响团队未来三个月资源投入的投资分析，而不是咨询报告或媒体稿。

要求：

- 第一段给判断
- 每个判断都有证据和不确定性
- 用短句、具体名词、主动语态
- 不使用营销词
- 不输出未经解释的缩写
- 每个核心技术词都要给白话解释
- 明确说出最大风险和下一步必须验证的证据

这个 skill 的价值不在于写得长，而在于把“看起来不错”和“真的值得继续投入”分开。

## 维护建议

维护这个 skill 时，优先更新：

1. 章节结构和投资判断口径
2. 医疗、眼科、监管和商业化证据标准
3. 中国落地路径的真实案例和判断模板
4. 常见术语的中文白话解释

不要把 broad sourcing 内容塞进这个 skill。它是单公司深度研究工具，不是公司发现工具。
